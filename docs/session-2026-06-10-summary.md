# Сессия 2026-06-10 (ночь + день) — полная сводка

Самая результативная сессия проекта: от выключенного устройства до полного
AVRCP-коммутатора. 12 коммитов (`2c3d30a … 4ed3f90+`), 23 пронумерованных
runtime-запуска (`artifacts/dual-mode-tests-20260610/`), все выводы — с
лог-доказательствами.

## Достигнутое состояние (всё работает одновременно, lab runtime)

- **Труба**: iPhone → Car Thing → Fosi, сквозной AAC без транскодирования,
  `forwarded≫0 dropped=0 sent_to_speaker=True`, музыка реально звучит.
- **Коммутатор AVRCP**: две параллельные сессии (iPhone + Fosi), никто никого
  не вытесняет.
- **Backchannel**: play/pause/next/prev с пульта Fosi управляют iPhone
  (пульт → AVRCP → мост → AMS-интент → iPhone).
- **Громкость iPhone → Fosi**: слайдер двигает усилитель (`SetAbsoluteVolume`).
- **Метаданные**: AMS живой (трек/плеер/позиция на каждое событие).
- Устойчивость: пауза/resume (минуты), Siri, смена треков, 20+ минут стрима,
  многократные рестарты runtime с автоподъёмом всего стека.

## Хронология и фиксы

### Ночь: восстановление и труба
1. Закоммичен staged-прорыв 5 июня (flush-фикс, `2c3d30a`) + ранбук с Apple ADG
   (`830046e`, ADG в `reference/apple-adg/`).
2. Бонд iPhone оказался забыт на телефоне → чистая BLE-first пара → CTKD выдал
   новый полный бонд (ltk+irk+link_key) — путь с нуля работает.
3. `PAGE_TIMEOUT` classic-дозвона: page гонится с BLE-реконнектом за одно
   радио → ретраи 3/8/15 c (`fc2cb5e`).
4. Headless enrollment Fosi: `CARTHING_PAIR_SPEAKER=<addr>` без инквайри/GUI.
   Fosi умеет AAC до 320k → сквозной AAC. **B1 зелёный** (музыка звучит).
5. B5 красный → 3 фикса (`23c5ba6`): suspend ≠ teardown (INVARIANTS п.3),
   role=speaker при пустом SDP-скане (иначе standby не звонит), гейт порядка
   старта (колонка раньше classic-дозвона). B5 повтор зелёный.
6. Атрибуция `peer=` во всех AVRCP-логах (`2c9b20a`) — после двух ложных
   выводов из-за неатрибутированных строк.

### День: AVRCP-коммутатор (тезис владельца «не в одно жало»)
7. **Четыре наслоённых корня** мёртвого Fosi-backchannel (`974566c`):
   - singleton `avrcp.Protocol` — слот занимал iPhone;
   - встречный connect к колонке = L2CAP-коллизия → пиру 3 c форы
     (Fosi сам открывает AVCTP через ~0.2 c после старта аудио);
   - ⭐ vendored Bumble РВАЛ соединение при mode mismatch (Fosi просит ERTM
     на AVCTP) → по спеке L2CAP: `UNACCEPTABLE_PARAMETERS` с контрпредложением
     Basic, peer повторяет Configure → канал открывается. Под защитой
     `check-bumble-vendor.py`;
   - `speaker_command_handler` не был подключён к мосту.
   Результат live: **кнопки пульта Fosi управляют iPhone**.
8. **Громкость** (`44c1b89`, `4ed3f90`): iPhone→Fosi работает; Fosi→iPhone
   невозможен на ZD3 — исчерпаны 4 механизма AVRCP + независимый эталон
   (родной стек macOS: пара Mac↔Fosi, A2DP-поток, poll громкости 2/с — ноль
   трафика от пульта). У ZD3 два независимых тракта громкости.
9. Семантика громкости маршрута: interim колонки съедается, источник — хозяин;
   «прыжок при коннекте» = пер-маршрутная память громкости САМОГО iOS
   (восстанавливает запомненный уровень, в логе он приходит ОТ iPhone) —
   штатно, самоизлечивается.

## Открытия / методология (переиспользовать)

- **Эталонный стек как арбитр** (идея владельца): спорный кейс «наш баг vs
  железо» решается подключением устройства к родному стеку Apple/BlueZ.
- **Подписка вопреки capabilities**: устройства врут в GetCapabilities
  (ZD3 не заявляет VOLUME_CHANGED, но регистрацию принимает).
- **Атрибуция пира в каждой строке протокольного лога** — обязательна.
- **Контролируемый эксперимент**: загрязнённый тест (run20) дал ложный сигнал;
  чистый (run21, iPhone не трогаем) — истинный.
- iOS: SDP-кэш, липкий выбор маршрута в CC, пер-маршрутная память громкости.
- ZD3: хранит несколько пар; мигает до открытия медиаканала; ERTM на AVCTP;
  команды успеха CONTROL = `ACCEPTED` (Bumble `_check_response` это не умеет).
- busybox: kill только по PID; runtime сам выходит после 8 неудач HCI.
