> Базовое состояние = точка невозврата **`2136ace` + тема `a1021e2`, p1 = vfat** (тег `base-pono-2136ace`).

## ЧАСТЬ 1. Реставрация недостающих фич (СДЕЛАНО — ветка `vfat-restore`)

Восстановлено поверх точки невозврата, на vfat, без единого ext4-файла. Worktree: `~/Documents/ПРОЕКТЫ/carthing-vfat-restore`. Каждая фича — отдельный закомментированный коммит. Проверено: весь рантайм `py_compile`, ноль `ext4`/`p3` во всём overlay, p1 = vfat.

| # | Коммит | Что вернули |
|---|--------|-------------|
| 1 | `4033c83` | тема «Терминал» + identity-registry (= состояние устройства) |
| 1b | `2e7720c` | правка лживого ext4-комментария → правда о vfat |
| 2 | `3949b79` | **режимы работы** (playnow/commutator/reserved) + **безопасное отключение** (`prepare_for_usb_unplug` вместо poweroff) |
| 3 | `60ee78d` | **accessory_orchestrator** — добавить вход не роняя выход |
| 4 | `94de096` | **надёжная USB-сеть** (S05/S04 — лечит «SSH виснет») |

**Статус:** НЕ собрано в образ, НЕ прошито. Ждёт решения после Части 2.
**Сознательно НЕ брали:** всё ext4/p3 (миграция state, etc/default→p3, S03/S11-ext4, defconfig, `tools/carthing-migrate-*`); реальный poweroff/halt (под предохранителем `CARTHING_ALLOW_LINUX_POWEROFF`).

## ЧАСТЬ 2. ПРИОРИТЕТ — изоляция GUI / Bluetooth / аудио по ядрам

### Проблема (проверено по коду, не по памяти)
Рантайм сейчас — **один процесс, один asyncio-loop** (`carthing_runtime.py: asyncio.run(main())`). В одном loop живут: **Bumble** (BT: paging/L2CAP/A2DP/AVDTP/AVRCP), **аудио-транскод** AAC→SBC (Helix decode + SBC encode) и **GUI-оркестрация**; GUI-рендер вынесен в executor-поток. Всплеск GUI-рендера + **GIL** «затыкает» loop, где живёт timing-критичный BT и транскод. Это правдоподобный корень открытых BT-деградаций (`claude-airfly-avdtp-degraded`, `claude-standby-reconnect`, PAGE_TIMEOUT).

**Голый `taskset` по ядрам внутри одного процесса не лечит** — один GIL, потоки не исполняют Python-байткод параллельно (выигрыш только на C-участках, отпускающих GIL).

### Что реально доступно на устройстве (конфиг ядра 4.9 Amlogic, проверено)
| Флаг | Значение | Вывод |
|---|---|---|
| `CONFIG_PREEMPT=y` | вкл | ядро уже полностью вытесняемое (low-latency). **PREEMPT_RT не нужен** |
| `CONFIG_CPUSETS=y`, `CGROUPS=y`, `CGROUP_SCHED=y` | вкл | **cpuset (runtime, без ребута) доступен** — современный предпочтительный путь |
| `CONFIG_HIGH_RES_TIMERS=y`, `PREEMPT_RCU=y` | вкл | хорошая база под soft-RT |
| `CONFIG_IRQ_FORCED_THREADING=y` | вкл | можно потоково обрабатывать IRQ + дать им RT-приоритет/affinity (`threadirqs`) |
| `CONFIG_NO_HZ_FULL` | выкл | полный tickless на ядре недоступен без пересборки → **не цель** |
| `CONFIG_HZ=250` | 250 Гц | tick 4 мс; для A2DP (буферизован) достаточно |

**Целимся в soft-RT** (A2DP имеет jitter-буфер — hard-RT не нужен). Всё достижимо **без пересборки ядра и без трогания загрузчика** (инвариант цел): cpuset + `SCHED_FIFO` + IRQ-affinity — всё runtime; `isolcpus` — опционально через редактируемый `bootargs.txt` (это boot-файл, не загрузчик).

### Целевая архитектура (принимаем ОДИН раз — «no refactor»)
Ресёрч подтверждает: при независимых задачах логику разносят на **отдельные процессы**, а синхронизируют через **shared memory** (низшая латентность; pipe — следом; TCP/unix-socket — последним).

1. **GUI → отдельный процесс.** Свой интерпретатор/GIL, свой рендер PIL→DRM. Читает `AppState` из **разделяемой памяти** (или unix-socket для команд). GUI физически не может заблокировать BT-loop.
2. **BT + аудио → процесс рантайма.** Горячие потоки (asyncio-loop BT, транскод-поток) — `SCHED_FIFO` (prio ~2–3 для аудио).
3. **Разделение ядер через cpuset** (runtime, обратимо):

| Ядро | Назначение |
|---|---|
| **0** | ОС, init, ssh, основная масса IRQ (`irqaffinity=0`) |
| **1** | **GUI-процесс** (рендер/компоновка) |
| **2** | **BT-рантайм** (Bumble loop), сюда же IRQ UART `/dev/ttyS1` (BT-чип) |
| **3** | **аудио-транскод** AAC→SBC (`SCHED_FIFO`) |

4. **IRQ-affinity:** IRQ BT-UART → ядро 2 (рядом с BT-loop); прочие IRQ → ядро 0.
5. **isolcpus (опционально, позже):** `isolcpus=2,3` в `bootargs.txt` для жёсткой изоляции RT-ядер — но сначала cpuset (без ребута, обратимо).

### Стадии внедрения
- **Стадия 0 — быстрая проверка гипотезы** (полчаса, без смены архитектуры): per-thread `os.sched_setaffinity` — рендер-поток → ядро 1, loop(BT+аудио) → ядра 2-3; замерить, падает ли BT-jitter. Подтверждаем/опровергаем корень.
- **Стадия 1 — настоящая изоляция** (целевая): вынести GUI в отдельный процесс + shared-memory IPC; cpuset-разметка ядер; `SCHED_FIFO` горячим потокам; IRQ-affinity BT-UART. **Это и есть «попугаи не мешают Bluetooth».**
- **Стадия 2 — максимум** (опц.): `isolcpus=2,3` + `threadirqs` + RT-приоритет IRQ-потоку BT.

### Открытые вопросы Стадии 1 (решить до кодинга)
- Формат shared-memory `AppState` (mmap-структура vs `multiprocessing.shared_memory`) — память устройства 512 МБ, доп. процесс ≈30 МБ (приемлемо).
- Кто владеет `/dev/dri`/framebuffer — GUI-процесс единолично.
- Запуск/надзор: два сервиса в init, рестарт независимо.
