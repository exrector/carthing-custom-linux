# Исследование микрофонного тракта Car Thing — 2026-06-24

**Цель сессии:** довести Car Thing до роли беспроводного микрофона для голосового
ИИ-ассистента (`~/Documents/ПРОЕКТЫ/voice-assistant`, faster-whisper `medium`,
язык `ru`). Жалоба владельца: «бешеная задержка», ассистент не понимает, где
конец фразы, разработка буксует.

Девайс под тестом: **Car Thing QN19**, `root@172.16.42.77`, Amlogic g12a,
4× Cortex-A53 @1.8 ГГц, 499 МБ RAM (~380 свободно), zram 128 МБ, python 3.14.

> ⭐ **ИТОГ В ОДНОЙ СТРОКЕ:** беспроводной мик достижим. Проблема была НЕ в
> Bluetooth и НЕ в качестве. Два дефекта на устройстве: (1) PDM-гейн в SoC стоял
> на слабой ступени → −29 dBFS; (2) кривой ресемпл (`maxabs` + дециммация «каждый
> 3-й») заливал паузы алиасинг-мусором → endpointing не ловил конец фразы.
> **Оба чинятся в SoC/на устройстве, доказано измерениями.**

---

## 1. Корректировки постановки задачи (что было неверно)

1. ❌ «Bluetooth даёт только телефонное 16 кГц» — неважно: потребитель Whisper,
   которому **16 кГц mono = нативный вход**. Качество не проблема.
2. ❌ «Проблема в полосе/чанкинге» — нет: 16к·mono·s16le = 256 кбит/с, ничто.
3. ✅ **Настоящая проблема — endpointing (границы высказывания).** Whisper ждёт
   *конца* фразы, чтобы распознать её целиком. Грязный непрерывный поток не даёт
   эндпоинтеру поймать «тишину» → высказывание не закрывается → Whisper получает
   безграничный блоб → галлюцинации.

## 2. Карта тракта (как есть)

```
Car Thing: PDM mic (pcmC0D1c, 48k/4ch)
   └─ session_plane_service._mic_loop → _downmix_resample_16k  ← ЗДЕСЬ РОЖДАЕТСЯ МУСОР
   └─ CTSP (кастомный BLE L2CAP CoC, frame T_AUDIO_PCM16=0x06)
        │  Bluetooth (надёжный, упорядоченный — НЕ источник мусора)
Mac: carthing-mic-agent (Mach-O Swift) → CoreAudio Loopback
   └─ audiocap.swift → wake_vosk.py (Silero VAD + адаптивный RMS endpointing)
   └─ stt.py (faster-whisper, ru, vad_filter=False, + большой _is_hallucination)
```

Прочие (не основной путь): `remote_mic_sender.py` (raw PCM по TCP/USB-NCM),
`hfp_handsfree_lab.py` (HFP/SCO — сломан, нет mSBC-энкодера, гонит сырой PCM как
SCO-кадры; HFP для этой задачи не нужен — приёмник наш агент, шлём данные по CTSP).

## 3. Что проверено на живом железе (и опровергнуто)

### 3.1 Внешнего чипа-усилителя микрофонов НЕТ (доказано 4 способами)
- **I2C-скан всех шин** (метод i2cdetect, EBUSY=занят драйвером): на шинах только
  touch `0x2e`, accel `0x18` (не отвечает), tmd2772 `0x39`, mfi `0x10`,
  **MAX20332 `0x35`** (USB-C audio/data мукс, не мик-усилитель). Аудио-чипа нет.
- **Драйвер `tlv320adc3101` загружен, но НЕ привязан**: `/sys/bus/i2c/drivers/
  tlv320adc3101/` пуст (только bind/unbind, ноль устройств).
- **Kernel-config:** TAS5805/TLV320ADC3101/SSM3525/PCM186X/AD82584F/CS42528 = `y`,
  но это BSP Amlogic скопом, все unbound. (`ДРАЙВЕРЫ-ЯДРА-И-ЖЕЛЕЗО.md`: «этих
  чипов нет на плате»).
- **Эксперимент `lr_gain` (tlv320adc3101) 32→80 = ноль эффекта** на захвате.
  Codex крутил этот `CARTHING_BT_MIC_LR_GAIN` (даже выставлял 64) — **knob
  несуществующего чипа**, делал ровно ничего.

### 3.2 T9015 — внутренний кодек SoC, но к микам не подключён
- `T9015 (AMLT9015)` — встроенный в SoC аналоговый кодек: ЦАП (line-out на
  `pcmC0D0p`) **и** аналоговый АЦП с PGA (`"PGA IN Gain"`, до ~+65 дБ в драйвере).
- **НО тест захвата с `pcmC0D0c` (T9015 HIFI Capture) = rms 1.0, peak 1** — мёртвая
  тишина. К аналоговому входу T9015 микрофон не подключён. Усиливать ему нечего.
- Микрофоны — **4 цифровых PDM MEMS, напрямую в SoC** (DTS: `PDM_DIN0→Left
  Microphones`, `PDM_DIN1→Right Microphones`, `pdm_codec="/dummy"`). У PDM-микрофона
  нет аналогового PGA физически.

### 3.3 Beamforming на 4 микрофонах — мёртв
Живой замер 4 каналов (под голос, самое громкое окно):
```
corr 0-1=0.998  0-2=0.997  0-3=0.992    задержка между миками: 0..1 сэмпл (≤0.02 мс)
```
Каналы — почти идентичные копии, разницы времени прихода нет → микрофоны
физически в одной точке (маленький массив). Пространственной диверсити нет →
вычесть «шумовой» канал из «голосового» нельзя (убьёт и голос). Гипотеза «одни
мики берут голос, другие шум» для ЭТОГО железа не работает.

### 3.4 `/dev/vad` (аппаратный VAD-DSP) — не поднимается
dmesg: `Register vad` → `Not init vad` → `Failed to add VAD controls`. Нода есть,
драйвер не инициализируется. Аппаратный low-power VAD сейчас недоступен.

## 4. ⭐ Что НАЙДЕНО и РАБОТАЕТ

### 4.1 Железный гейн микрофонов = `PDM HCIC shift gain` (в SoC)
ALSA-kcontrol, перечислен через control-ioctl (на девайсе нет `amixer`/`tinymix`):
```
numid=6  enum 2 items  'PDM HCIC shift gain from coeff'
```
Sweep по ступеням (живой замер уровня PDM):
```
gain_index=0 (наш дефолт):  rms=135.0  peak= 878    ≈ −29 dBFS  ← почему мик «тихий»
gain_index=1:               rms=876.2  peak=5373    ≈ +16 дБ, ×6.5, БЕЗ клиппинга
```
Наш кастомный кейрнел держал гейн на слабой ступени (index 0). Сток Spotify дёргал
ALSA-контролы через `amixer` (заводской `S49amixer` ставит `TODDR_A SRC SEL=IN 4`).
У нас amixer нет → дефолт. **«Hey Spotify!» на этом железе работал именно потому,
что гейн был настроен.** Флип на index 1 = +16 дБ чистого железного усиления.

### 4.2 Перенаправление потока через аппаратный ресемплер SoC
```
numid=13  enum 8 items  'Hardware resample enable'  cur=0 (выключен)
numid=16  enum 16 items 'Hw resample module'        cur=4 = PDMIN  ← УЖЕ нацелен на мики
```
В `enum toddr_src` есть `PDMIN=4`. Ресемплер уже наведён на PDM, просто выключен.
Включение → чистый железный 48→16 с анти-алиасингом, без питоновской дециммации.

EQ/DRC (AED) для makeup-gain на захвате — **нельзя**: `aed_req_module_texts` =
только выходы (`TDMOUT_A/B/C`, `SPDIFOUT_A/B`). Блок односторонний (тюнинг динамиков).

### 4.3 Корень «не ловит конец фразы» = `_downmix_resample_16k`
`overlay/usr/lib/carthing/session_plane_service.py`, метод `_downmix_resample_16k`:
- `mix_mode="maxabs"` (дефолт) — посэмплово берёт самый громкий из 4 каналов →
  нелинейная сшивка кусков разных миков → щелчки/разрывы → широкополосный шум.
- `step = round(48000/16000) = 3`, `for frame in range(0, n, step)` — **наивная
  дециммация без анти-алиаса**: всё из 8–24 кГц зеркалится в речевую полосу как
  постоянный шипящий пол → «тишина» никогда не наступает → эндпоинтер
  (`wake_vosk.py`: Silero VAD + адаптивный RMS) не закрывает фразу → отсюда же
  огромный костыль `_is_hallucination` в `stt.py`.
- `int(sample*gain)`, gain=2.0 с хард-клиппингом — добавляет гармонический мусор.

## 5. ⭐ Финальный тест (голос владельца, HCIC gain=1)

Записано 10 с сырья 48k/4ch (peak~13053 ≈ −8 dBFS — против −29 dBFS без гейна),
обработано двумя способами в 16k mono:
```
                     peak    громко   ТИШИНА(пауза)   динамика
CODEX maxabs+dec3    17442    5355       174.6         30.7x
CLEAN avg+antialias  15633    4887        96.6         50.6x   (scipy.resample_poly)
```
**Выводы:**
1. Железный гейн HCIC=1 → нормальный уровень (−8…−6 dBFS), без клиппинга. Решает
   «слабый мик» в железе.
2. Чистый ресемпл → пол в паузе −45 % (174→96), динамика +65 % (30→50×). Алиасинг,
   заполнявший тишину, ушёл → endpointing получает настоящую тишину.
3. Остаток ~96 в паузе — **реальный фон комнаты**, не алиасинг. Добивается
   нойз-гейтом/одноканальным денойзом (RNNoise) на Mac.

Артефакты теста: `mic_codex.wav` / `mic_clean.wav` (различимо на слух в паузе).

## 6. Решение и план внедрения

**Беспроводной мик достижим. Транспорт — данные по CTSP (не HFP).** Усиление —
цифровое в SoC (природа PDM; сток Spotify тоже усиливал в софте через Sensory AGC).

1. **Устройство — гейн:** в `session_plane_service._mic_loop` ставить
   `PDM HCIC shift gain`=1 через control-ioctl `/dev/snd/controlC0` перед захватом
   (helper-структуры см. ниже). Это +16 дБ в SoC, до квантизации в int16.
2. **Устройство — ресемпл:** заменить `_downmix_resample_16k` (maxabs+dec3) на
   **усреднение 4 каналов + анти-алиас ресемпл 48→16** (как `clean` в тесте), ЛИБО
   включить аппаратный ресемплер (`numid=13`, module уже=PDMIN). Убрать слепой
   `gain*2` с клиппингом.
3. **Mac — добить фон:** нойз-гейт/RNNoise в `audiocap`-пайплайн на остаточные ~96.
   После чистого источника существующий Silero/RMS endpointing должен закрывать
   фразы; костыль `_is_hallucination` в `stt.py` можно урезать.

### Воспроизводимость: чтение/запись ALSA-kcontrol без amixer
`amixer`/`tinymix` на rootfs нет. Контролы дёргаются control-ioctl на
`/dev/snd/controlC0` (структуры `snd_ctl_elem_{list,info,value}`, aarch64):
- `SNDRV_CTL_IOCTL_ELEM_LIST` = `_IOWR('U',0x10,sizeof(elem_list))` — перечислить.
- `..._ELEM_READ` = `0x12`, `..._ELEM_WRITE` = `0x13` (sizeof(elem_value)=1224).
- enum: значение = индекс в `value[0]` (LE). Рабочий скрипт-helper отработан в
  сессии (перечисление + sweep гейна + восстановление исходного).

Релевантные контролы (card0): `numid=6` PDM HCIC shift gain (0/1),
`numid=13` Hardware resample enable, `numid=16` Hw resample module (=4 PDMIN),
`numid=5` PDM Filter Mode, `numid=8` PDM Train.

## 8. Пространственный шумодав / beamforming — проверено ушами (дополнение)

Раздел §3.3 закрывал beamforming по тесту с источником **по центру** (corr 0.998).
Это методологически слабо: центральный источник и должен давать ~0 задержки. Провели
**правильный** тест — направленные источники: **музыка слева, голос справа**.

**Грабли (важный урок):** первый дубль записал **пустой фон** — окно стартовало раньше,
чем владелец начал говорить (RMS ~800 = усиленный ×6.5 тихий фон, не музыка/голос).
Анализ на нём (coherence 0.20, NLMS 23.8 dB) был **невалиден**. Правило: перед выводами
**проверять, что событие реально в записи** (по-секундный лог уровня / прослушать WAV).

**Валидный тест** (20 с, по-секундный лог подтвердил живой контент, гейн=1):
```
ch0=892 ch1=895 ch2=860 ch3=856        уровни почти равны
L-pair(ch0,1)=885   R-pair(ch2,3)=850  пары слышат ОДИНАКОВО громко
corr(L,R)=0.915                        упало с 0.998, но высоко
rms(L-R)=360                           пространственный остаток ~40% амплитуды
```
**Вердикт владельца на слух:** `left_pair` и `right_pair` звучат **почти одинаково**.

**ИТОГ: пространственного NES на этом массиве не выжать.** Мики слишком близко (мелкий
массив, задержка ~0), направленности нет. Спатиальная диверсити есть, но слабая
(corr 0.915) и для нас не эксплуатируется тривиально. Стоковый «Hey Spotify» вытягивал,
вероятно, не beamforming'ом, а **AEC против собственной музыки** (reference известен) +
шумоустойчивой wake-word моделью (всё в вырезанном Sensory DSP).

**Что 4 мика реально дают нам:** усреднение 4 каналов = **−6 дБ собственного электронного
шума** (√4). Используем их как «4 мика → один хороший моно-мик». Спатиально — нет.

## 9. ⭐ Финальная архитектура (одноканальная)

```
Девайс:  4ch → усреднить (−6дБ шум) → HCIC gain=1 (+16дБ) → чистый ресемпл 16k → CTSP↑
Mac:     AEC (против своего TTS/музыки, reference известен — ГЛАВНЫЙ кейс ассистента)
         → RNNoise (одноканальный денойз фона комнаты)
         → AGC → faster-whisper (ru) → текст ответа → CTSP↓ → экран Car Thing
```
- **STT только на Mac.** RAM устройства = **512 МБ** (498 МБ, ~414 свободно; CMA 225 МБ
  частично отбираема). `faster-whisper medium` (~1–2 ГБ) не лезет. «4 ГБ» = это **eMMC
  хранилище (3.69 ГиБ)**, не RAM — частая путаница.
- **Ответ возвращается ТЕКСТОМ на экран** (у девайса нет колонки). CTSP — двусторонний
  канал данных (аудио↑, текст↓), не аудио-handset. Добавить тип кадра «текст ответа».
- Шумодав-приоритет: **AEC по своему звуку** (надёжно, reference есть) > RNNoise
  (произвольный фон) > массив (отклонён). AGC решает «дальний угол» лучше, чем фикс-гейн.

**Надёжно в руках (проверено):** HCIC gain=1 (+16дБ, без клиппинга), чистый анти-алиас
ресемпл (пол в паузе −45%, динамика +65%), усреднение 4ch (−6дБ электронного шума).

## 10. ⚠️ ПОПРАВКА к §8: массив пространственно НЕ мёртв (по-частотный замер)

Вывод §8 «спатиального NES не выжать» был **преждевременным** — основан на грубой
**широкополосной** корреляции (0.915) и грубом тесте «пары на слух». Это вводило в
заблуждение: почти вся энергия в низах, где длинная волна → мики и должны коррелировать.
Правильное измерение — **по-частотная когерентность на валидной записи** (`mic_spatial2`):

```
pair     50-300 300-1k 1k-2k 2k-4k 4k-8k 8k-16k 16k-24k
ch0-ch3   0.93   0.79  0.47  0.63  0.49  0.53   0.28
ch1-ch2   0.93   0.96  0.85  0.78  0.45  0.55   0.31
задержка соседних миков ch0-ch1/ch2-ch3/ch1-ch2 = −0.7..−0.8 сэмпла (~6 мм, стабильно)
```

**Это подпись реального пространственного массива:** когерентность плавно падает с
частотой (0.9→0.3); в речевой полосе 1–8 кГц каналы несут разную инфу (0.45–0.6). Будь
это шум АЦП — когерентность была бы низкой и **плоской**, а она спадающая = **геометрия**.
Teardown подтверждает: **4 MEMS вдоль верхней грани (~10 см), 4 отдельных отверстия**.

**ИСПРАВЛЕННЫЙ вывод:** массив — разнесённый дальнобойный beamforming-массив,
пространственная диверсити есть и **эксплуатируема** (мид/высокие частоты). Стоковый
«Hey Spotify» вытягивал именно им (+ AEC против своей музыки) через Sensory DSP.
Путь воспроизвести без Sensory: **WebRTC APM** с геометрией миков, либо свой beamformer.
Одноканальный RNNoise — это fallback/упрощение, а НЕ «потому что массив бесполезен».

**Урок методологии:** широкополосная корреляция и «на слух» — НЕ доказательство по
пространству. Только по-частотная когерентность + проверка, что событие реально в записи.

## 11. ⛔ ТУПИК: попытка запустить стоковый AudioWeaver DSP на устройстве (НЕ ПОВТОРЯТЬ)

**Решение владельца 2026-06-25: шумодав на Linux/устройстве НЕ делаем. Точка.** Устройство
только усиливает мик железом (HCIC gain) и шлёт чистый поток. **Шумодав — Apple
VoiceProcessing на macOS** (AEC+NS+AGC, уже в `voice-assistant/audiocap.swift`).

### Что я нагородил (зря потраченные шаги — чтобы не повторять)
Пытался поднять заводской **AMLogic AudioWeaver** DSP (beamformer+NS+AEC+Sensory wake) на
устройстве. Проделано (всё снесено после):
1. Нашёл в бэкапах vendor-пример `brcm-bsa/.../libdspc`: движок `libAWELib.so` (ARM32, 854
   модуля, есть `SbBeamformer`/`SbAECV1`), `.awb` (`AMLogic_VUI_Solution_v6a_VoiceOnly`).
2. Кросс-собрал ARM-харнесс (clang `--target=arm-linux-gnueabihf` + linaro sysroot из
   бэкапов + `lld`), залил armhf-рантайм + движок на устройство (ядро `CONFIG_COMPAT=y`).
3. Запустил: `LoadAwbFile` падал `-50` пока не доложил `netfile.raw/searchfile.raw` (Sensory
   модель — wake-слово оказалось **«Alexa»**, демо AMLogic, не «Hey Spotify»).
4. Долго путался в раскладке каналов. Итог из боевого `xueming1988/test1`
   `soft_prep_dspc_func.cpp` (`g_dspc_mic_array_4`): 4 мика → AWB-каналы **0,2,4,6**
   (не 0-3 и не 2-5), feedback(AEC) → 8,9, выход `>>14`.
5. С верными каналами пайплайн **выдал непрерывный выход** — но **шумодава на слух нет**
   (без правильной геометрии/направления луча и без AEC-reference это не работает).

### Почему тупик
- AWE Designer (правка layout'а) — **только Windows, которого у владельца НЕТ** ([[feedback_no_windows]]).
- Beamforming требует точной геометрии массива + наведения луча (в стоке — Sensory DSP,
  вырезан). Воспроизвести вслепую = спираль галлюцинаций. **Я несколько раз выдавал догадку
  за вывод** («wake-gate», «массив мёртв») без проверки — ровно [[feedback_never_half_ass]].

### Что РЕАЛЬНО осталось в проде (минимум, проверено)
- `session_plane_service._set_device_mic_gain()` — ставит **PDM HCIC shift gain = 1** (+16 дБ,
  железный гейн SoC) через ALSA control-ioctl. Это «точка опоры».
- НЕ трогать каналы заново (они выверены), НЕ городить Linux-DSP, НЕ предлагать Windows-тулы.

## 12. ✅ РЕШЕНИЕ: шумодав = DeepFilterNet на Mac (нейросеть, ~уровень Apple Voice Isolation)

«Шумодав на Linux нельзя» — **было неверно** (мой провал, владелец нашёл за минуту). Есть
RNNoise / **DeepFilterNet** / WebRTC APM — нейросетевые open-source шумодавы. Выбран
**DeepFilterNet3** (лучшее качество, `pip install deepfilternet`, Python — под ассистент).

**Боевой тракт:**
```
Car Thing:  PDM-мик → HCIC gain=1 (+16дБ железом) → avg 4ch → 16k → поток
Mac:        DeepFilterNet3 (denoise.py) → Whisper (stt.transcribe) → ассистент
```

Реализация:
- `voice-assistant/denoise.py` — обёртка DeepFilterNet: 16к→48к→enhance→16к, ленивая загрузка,
  потокобезопасно, `VA_DENOISE=0` выключает. Шим для `torchaudio.backend.common` (DFN 0.5.6
  vs новый torchaudio). Установка откатывает numpy 2.4→1.26 — стек цел (проверено).
- `voice-assistant/stt.py` `transcribe()` — вызов `denoise.enhance_16k(audio)` перед Whisper.
- `voice-assistant/main.py` — Apple VoiceProcessing выключен (`VA_DISABLE_VOICE_PROCESSING=1`),
  шумодав делает DeepFilterNet (Apple конфликтовал с Loopback + двойная NS).

**Живые прогоны с Car Thing-мика (голос владельца, счёт + фон):**
```
тихо:   пол паузы raw=0.0227 → clean=0.0002 (−41 дБ)
шумно:  пол паузы raw=0.0550 → clean=0.0001 (−55 дБ);  Whisper распознал верно
скорость: ~0.9с на 8с аудио (CPU, ~9× realtime)
```
Результат подтверждён на слух владельцем: «потрясающие результаты».

## 13. ✅ ЖИВОЙ GUI-ассистент целиком (ЧАСТЬ 1 — транспорт USB)

Полный цикл проверен вживую с разных углов комнаты/расстояний (владелец): голос → шумодав →
распознавание → ответ LLM (mistral) → голос + диалог в GUI `VoiceAssistant.app`.

**Тракт (часть 1, транспорт = USB/NCM 172.16.42.77):**
```
Car Thing PDM-мик → HCIC gain=1 → mic_server :49400 (устройство, персистентный TCP)
  → carthing_audiocap.py (Mac, TCP, 4ch→моно float32) → audiocap-источник ассистента
  → DeepFilterNet (denoise.py/stt.py) → Whisper → mistral → TTS + GUI
```

**Файлы (в `voice-assistant/`):**
- `carthing_mic_server.py` — деплоится на устройство в `/tmp/mic_server.py`: HCIC gain=1 +
  персистентный захват pcmC0D1c, отдаёт сырьё 4ch int16 по TCP :49400 (один захват, ALSA
  эксклюзивен; `settimeout` — не клинит на мёртвом клиенте при рестартах audiocap).
- `carthing_audiocap.py` — audiocap-источник: быстрый TCP-connect к :49400, 4ch→моно float32 @48k.
- `denoise.py` — DeepFilterNet3 шумодав.
- `run_carthing.sh` — поднимает всё: деплой+старт mic_server, затем ассистент
  (`./run_carthing.sh --gui` для окна, без флага — headless).

**Грабли (решены):** ассистент **рестартит audiocap каждый цикл** → SSH-на-спавн не годится
(старт ~2с) → персистентный TCP mic_server + быстрый connect. Два mic_server = конфликт ALSA
(только один!). После kill — `sleep 2` на освобождение ALSA перед рестартом.

**Задержка ответа** — ассистент-сайд (Whisper-medium STT ~3с + первый токен LLM по сети + TTS),
не тракт мика. Лечится меньшей STT / локальным-быстрым LLM / стримингом TTS.

## 14. ✅ ЧАСТЬ 2 — BLE L2CAP CoC канал device↔Mac ПОДНЯТ (2026-06-25)

**Канал работает.** Mac (`CarThingBTAudioCap`) подключается к Car Thing по BLE и стримит мик
по L2CAP CoC. Подтверждено device-логом:
```
connected <MAC>/P classic=False → session peer LE attached → session CoC connected psm=128
→ session client_toggle via GATT -> True → session command 'start_mic' → remote mic started
→ session remote mic hardware gain set
```

### Что было реальным блокером (не то, что я думал)
- НЕ TCC на Mac (бинарь имеет встроенный Info.plist с NSBluetoothAlwaysUsageDescription —
  Package.swift `-sectcreate __info_plist`; запускается без промпта).
- НЕ «BLE-транспорт не реализован» — он реализован с обеих сторон.
- **Реальный блокер: устройство не рекламило session-сервис C7C5.** Реклама C7C5
  (`COMPLETE_LIST_OF_128_BIT_SERVICE_CLASS_UUIDS = SESSION_SERVICE_UUID`) включается в
  `_adv_payload()` ТОЛЬКО при `session_advertising=True`, а это = **Client ON**.

### КАК ПОДНЯТЬ (рабочая процедура)
1. **Device — Client ON:** `settings.client_enabled=True` в
   `/run/carthing-state/carthing/state.json` (это **персистентный vfat-раздел mmcblk0p1** —
   переживает ребут). На boot `_on_toggle_client(True)` → `set_session_advertising(True)` →
   `_adv_payload()` включает C7C5 → устройство видно Mac-скану по service-UUID.
   - usb_fallback OFF (`CARTHING_USB_REMOTE_MIC_ENABLE` НЕ задан) → транспорт `ble_l2cap_coc`.
2. **Mac:** запустить `macos-observer/.build/debug/CarThingBTAudioCap` (headless авто-коннект:
   scan по C7C5 → connect → GATT bootstrap → PSM → openL2CAPChannel → `start_mic`). Single-peer
   (один инстанс держит сессию).
3. Архитектура: device = BLE Peripheral + GATT server + L2CAP listener (psm=128, через
   `accessory_orchestrator` + `HciOperationGate` сериализует HCI); Mac = Central + L2CAP opener
   (`TransportCore`: CBCentralManager scan→connect→GATT→openL2CAPChannel→CoC). Протокол CTSP
   (framed, см. `Sources/ProtocolCore/CTSPFrame.swift` + `session_plane_service.py`).

### ⚠️ Грабли, которые я создал (НЕ повторять)
- **НЕ убивать `carthing-btattach-mini`** — он держит `hci0` (HCI-сокет для Bumble через
  `hci-socket:0`). Убил → рантайм крашится `BrokenPipeError` в `bumble/transport/hci_socket.py`
  → супервизор в петле. Краши `client_enabled=True` были ИЗ-ЗА этого, не из-за флага.
- Рестарт рантайма: `kill` python НЕ респаунит (обёртка `run-media-remote` виснет). Правильно:
  kill супервизора (`/run/carthing/media-remote-supervisor.pid`) + rm pidfile + kill рантайма/
  обёртки (НЕ btattach), затем `/etc/init.d/disabled-S50-carthing-remote start`. hci0 поднимает
  `/etc/init.d/S20-bt-init`.
- Счётчики процессов через `grep carthing_runtime` дают **самоматч** (cmdline grep содержит
  паттерн) — фильтровать `/proc/` и матчить точную строку.
