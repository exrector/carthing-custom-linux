## 1. Что Linux РЕАЛЬНО видит

```
mmcblk0       — 3.77 ГБ, вся eMMC
mmcblk0p1     — 32 МБ,  FAT32   → /run/carthing-state
mmcblk0p2     — 3.6 ГБ, ext4    → / (rootfs, но fs внутри = 485 МБ)
```

## 2. Что находится между p1 и p2 (невидимый зазор)

Между концом p1 (сектор 73727) и началом p2 (сектор 352256) — **278 528 секторов = 136 МБ**.
Этот зазор НЕ является Linux-разделом. Он доступен только как raw /dev/mmcblk0.

Внутри зазора — Amlogic APT-слои:

| Сектор   | Имя (Amlogic)   | Размер | Что там    |
|----------|-----------------|--------|------------|
| 73728    | reserved        | 64 МБ  | Amlogic APT + **unifykeys** (BT MAC, device keys) — **НЕ ТРОГАТЬ** |
| 237568   | env             | 8 МБ   | U-Boot env (flash.py пишет сюда через saveenv) |
| 270336   | fip_a           | 4 МБ   | Amlogic FIP slot A |
| 294912   | fip_b           | 4 МБ   | Amlogic FIP slot B |
| **319488** | **logo**      | **8 МБ** | **Наши загрузочные заставки (bootlogos.bin)** |

Logo доступен из Linux как `/dev/mmcblk0` (raw) или через Amlogic APT как `/dev/mmcblk0p7`.

## 3. Содержимое p1 (/run/carthing-state)

```
Image           16 МБ  — ядро Linux 4.9.113 (U-Boot загружает при старте)
bootargs.txt    552 байт — kernel cmdline (U-Boot загружает при старте)
superbird.dtb   105 КБ — Device Tree Blob
carthing/       — наш state (state.json, keys.json, BT-бонды)
._Image         4 КБ   — macOS-мусор (AppleDouble metadata)
._bootargs.txt  4 КБ   — macOS-мусор
._superbird.dtb 4 КБ   — macOS-мусор
.fseventsd/     —       — macOS-мусор (FSEvents journal)
```

p1 = dual-use: U-Boot читает ядро отсюда до загрузки Linux; Linux монтирует сюда же state.
macOS-мусор (._* и .fseventsd) попал при прямой записи файлов через macOS. Не критично.

## 4. Состояние p2 (rootfs)

Партиция: **3.6 ГБ** (сектора 352256–7733247).
ext4 filesystem внутри: **485.6 МБ** (наш rootfs.img = 512 МБ, после mkfs.ext4 — 485.6 МБ usable).
Занято: **73.6 МБ (15%)**.
Свободно в ext4: **408 МБ**.

Остальные **~3.1 ГБ партиции за пределами filesystem** — неиспользуемое пространство.
Теоретически доступны через `resize2fs` — но см. раздел 6 ниже.

---

## 5. Почему диаграмма 18 разделов была неверной

Показанная ранее диаграмма (p01 bootloader … p18 data) — это **Amlogic APT**:
внутренняя таблица разделов, хранящаяся в `reserved`-зоне (сектор 73728).
U-Boot и сам Amlogic её знают. Linux — нет.

Когда ThingLabs записал свой `bootfs.bin` с MBR-таблицей в сектор 0, MBR определил
только 2 партиции (p1 FAT32 и p2 Linux). Ядро Linux видит именно MBR — и только 2 раздела.

**Вывод: «Android-мусор» (boot_a/b, system_a/b, vbmeta, misc, data) не существует
как Linux-партиции. Они никогда не существовали на уровне Linux. Убирать нечего.**

---

## 6. «Ловушка resize2fs» — почему не делаем

Предложение расширить ext4 с 485 МБ до 3.6 ГБ через `resize2fs` на первый взгляд
выглядит привлекательно, но является ловушкой:

1. **Нет актуальной задачи.** Занято 15% (73.6 МБ), свободно 408 МБ.
   Ни одна задача из ПЛАН-ПРИОРИТЕТЫ не требует больше места в rootfs.

2. **Ломает bake-процесс.** `bake-rootfs.py` работает с `rootfs.img` фиксированного
   размера (512 МБ). После resize на устройстве — следующая прошивка перезапишет
   расширенную FS стандартным 512 МБ образом.

3. **Online resize на RO-rootfs.** Rootfs смонтирован RO (`ro,relatime`).
   Нужен remount RW → риск рассинхронизации состояния.

4. **Меняет SHA256 rootfs.img** — образ и устройство расходятся, теряем reproducibility.

**Правило:** расширение rootfs — только если задача явно требует > 400 МБ свободного
места. Сейчас таких задач нет.
