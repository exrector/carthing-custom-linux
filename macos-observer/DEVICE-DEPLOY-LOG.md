# DEVICE-DEPLOY-LOG — что Claude кладёт/запускает на устройстве Car Thing

Отдельный append-only журнал ВСЕГО, что попадает на само устройство (не на Mac):
файлы, их путь на устройстве, источник на Mac, зачем, как откатить, какие
процессы запускались и как их гасить. Цель — чтобы ничего не осталось мусором
или висящим фоновым процессом, и чтобы Codex видел полную картину.

**Правила для этого лога:**
- Каждый файл на устройстве — с шапкой-комментарием «ВРЕМЕННЫЙ ТЕСТ КЛИЕНТ-СЕРВЕР (Claude, <дата>)».
- Каждый запуск процесса — фиксировать PID/как убить и убивать после теста.
- Не занимать HCI `/dev/ttyS1`, пока на нём работает роутер Codex.
- Не трогать рабочие файлы Codex; класть только в выделенный путь (см. ниже).

## Выделенный путь на устройстве для тестовых файлов Claude

```
/tmp/claude-ctsp-test/        # эфемерно, чистится при ребуте — предпочтительно
```
(если нужен persistent — согласовать отдельно; по умолчанию только /tmp)

## Журнал

### 2026-06-18 — разведка перед деплоем (read-only)
Статус: идёт. Цель — установить факты до любого деплоя:
- доступность устройства (SSH/USB-network);
- какие процессы запущены, кто держит `/dev/ttyS1` (HCI);
- состояние BT (роутер Codex активен?).
Деплой и запуск listener'а — ТОЛЬКО после оценки, что не конфликтует с HCI.

Результаты разведки (read-only, ничего не занято):
- SSH `carthing` (172.16.42.77, usb0) доступен. Устройство QN19, USID `8555R08SQN19`,
  ядро 4.9.113 NixOS.
- BT-чип привязан к ядру как `hci0` через `carthing-btattach-mini` (PID 361) +
  kernel threads `[hci0]` (364/365). Контроллер ВИДЕН в `/sys/class/bluetooth/hci0`.
- Рабочий процесс Codex: `carthing_runtime.py` (PID 1110) — держит сокеты, пользуется
  единственным контроллером. `reverse-agent.py` (PID 323) — reverse-control мост.
- ВЫВОД: второй BLE-стек на hci0 = поломка сессии Codex. Радио НЕ трогаем.
  Device-side тест делаем по TCP поверх usb0.

### 2026-06-18 — деплой CTSP TCP-сервера (TCP/usb0, БЕЗ hci0)
Файлы на устройстве (все в эфемерном /tmp, чистятся ребутом):
| Путь на устройстве | Источник на Mac | Зачем |
|---|---|---|
| `/tmp/claude-ctsp-test/ctsp_test_server.py` | `macos-observer/device/ctsp_test_server.py` | эталонный device-side CTSP-сервер по TCP |
| `/tmp/claude-ctsp-test/server.log` | (рантайм) | stdout сервера |
| `/tmp/claude-ctsp-test/server.pid` | (рантайм) | PID для останова |

Управление: `macos-observer/scripts/device-ctsp-test.sh {deploy|start|status|stop|clean|verify-no-bt}`.

Прогон (Mac-клиент `swift run CTSPProbe 172.16.42.77 7777` ↔ device-сервер):
- connect → capabilities (`device:QN19`) → route_state OK;
- RTT hello = 4.32 мс (реальная usb0-сеть);
- on-demand mic: ~41 кадр/с ~27 КБ/с, RMS 0.21 (синтетический тон принят);
- 78720 б PCM ≈ 2.46 c; route получен.
- verify-no-bt: сервер НЕ держал ttyS1/bluetooth сокетов.

Очистка/состояние ПОСЛЕ теста (проверено):
- сервер остановлен (PID 1425 мёртв, порт 7777 свободен, в `ps` нет);
- `carthing_runtime.py` (1110) РАБОТАЕТ, `hci0` на месте, `btattach` (361) РАБОТАЕТ —
  сессия Codex не задета;
- файлы `/tmp/claude-ctsp-test/` ОСТАВЛЕНЫ для повторных прогонов (в /tmp, эфемерно).
  Полностью убрать: `scripts/device-ctsp-test.sh clean`.
