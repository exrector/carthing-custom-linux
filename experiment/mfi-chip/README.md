# MFi auth-чип Car Thing — clean-room реверс-инжиниринг (результат)

**Что это:** первый известный clean-room доступ к Apple MFi authentication-копроцессору внутри Spotify Car Thing (Superbird) — без закрытого userspace Spotify, без BlueZ, без `bluetoothd`, без D-Bus. Чип был заперт: штатный драйвер `apple_mfi_auth` в кастомной системе не биндится, `/dev/apple_mfi` отсутствует. Доступ восстановлен **напрямую по сырой I2C-шине**, протокол вскрыт, и **живьём доказаны извлечение сертификата и подпись challenge**. Поверх построен рабочий clean-room MFi/iAP2 auth-бэкенд.

Это готовый результат тестов — **не инструкция «как повторить»**. Здесь то, что нужно, чтобы пользоваться чипом из кода.

---

## Идентификация чипа

| | |
|---|---|
| Шина | `/dev/i2c-3` |
| Адрес | `0x10` |
| DT compatible | `apple_mfi_auth` (`/sys/bus/i2c/devices/3-0010`) |
| Семейство | Apple **ACP 3.0** (Authentication Coprocessor 3.0) |
| Версия (из чипа) | device version `0x07`, auth revision `0x01`, protocol `3.0` |

Драйвер в кастомной системе **не нужен**: всё делается из userspace через `/dev/i2c-3`. Если `/dev/apple_mfi` присутствует — можно и через него (ioctl-контракт ниже), но это не обязательно.

## Карта регистров (ACP 3.0, подтверждено живьём)

| Рег | Назначение |
|---|---|
| `0x00`/`0x01` | wake/prepare (короткие записи; меняют фазу) |
| `0x05` | error code (`0x00` = ок) |
| `0x10` | control/status (триггер `0x01`; готово когда читается `0x10`) |
| `0x11` | длина ответа (подпись → `0x0040`) |
| `0x12` | данные ответа (64-байтная подпись) |
| `0x20`/`0x21` | challenge length / **challenge data** (канонический вход) |
| `0x31` | старт стриминга сертификата |
| `0x4E` | легаси-триггер подписи (только диагностика, см. ниже) |

Фазовые байты после prepare: `07 01 03 00` = version `0x07` / auth-rev `0x01` / protocol major `0x03` / minor `0x00` → это и есть ACP 3.0.

## Сертификат устройства (доказано)

- Поток: `prepare` → короткая запись `0x31` → чтение 16-байтными чанками.
- Результат: **608 байт**, начинается с `30 82 02 5b` — валидный ASN.1 DER.
- `openssl asn1parse -inform DER` распознаёт как **PKCS#7 `signedData`**.

## Подпись challenge (доказано, ECDSA over prehashed SHA-256)

Канонная транзакция:
1. `prepare`: короткие записи `0x00`, затем `0x01`
2. записать `0x21` + **32 байта** challenge (это уже готовый SHA-256 digest)
3. триггер: записать `0x10 0x01`
4. поллить `0x10`, пока не станет `0x10` (ранний `nack` — норма)
5. прочитать `0x05` (error=`0x00`), `0x11` (len=`0x0040`), `0x12` (**64 байта подписи**)

**Реальный результат теста** — challenge `000102…1e1f` →
```
подпись (64 байта):
0164d52a60fb39e316c1cbbe77fafaa7ad73b6b91f160437323674835d5157d5
42313191b26ad4ae1843949151fad417d86dc9f497a43dd563ad2374da3a12fc
```
Проверяется как **ECDSA по предхешированному SHA-256** (challenge повторно НЕ хешировать).

### Про `0x4E` (важно для тех, кто будет копать дальше)
`0x4E` — НЕ самостоятельная команда подписи. Это временное окно совместимости поверх состояния, залатченного канонической записью `0x21`: подписывает **последний** challenge из `0x21`, живёт ~3-4 сек простоя, очищается cert-операциями. Использовать **только** канонический путь `0x21`; `0x4E` — диагностика.

## Как пользоваться из кода

Готовый инструмент: **`carthing-mfi-probe`** (в этой папке). Зависит только от `/dev/apple_mfi` или `/dev/i2c-3`.
```
carthing-mfi-probe info        # версия
carthing-mfi-probe certlen     # длина сертификата
carthing-mfi-probe response    # PKCS#7 сертификат -> stdout
carthing-mfi-probe sign <challenge_hex>   # 64-байтная подпись
carthing-mfi-probe aa01-live   # сертификат, завёрнутый в iAP2 AA01
carthing-mfi-probe aa03 <challenge_hex>   # подпись, завёрнутая в iAP2 AA03
carthing-mfi-probe auth-reply-live        # отвечает на AA00/AA02 прямо с чипа
```
Рекомендованная архитектура бэкенда — одна самодостаточная транзакция на запрос (всегда заново писать `0x21`, не полагаться на залатченное состояние), доступ к чипу сериализовать (cert и sign делят внутреннее состояние). Полный implementation-handoff — в `LAB-NOTES.md`.

## Что построено поверх (clean-room iAP2)

На проверенном cert+sign бэкенде поднят минимальный clean-room iAP2-стек (без BlueZ/Spotify): auth-ответы `AA00/AA01/AA02/AA03/AA05`, идентификация `1D00→1D01`, SDP-запись `Wireless iAP` (UUID `…caff`, RFCOMM ch 3), L2CAP PSM `0x0001`, HCI scan/EIR/SSP, Class of Device `0x240420` (Car Audio). Граница, найденная экспериментом: пользовательский pairing в Settings iPhone — это **HID-контракт**, а не сырой iAP2-транспорт (их нельзя смешивать).
